## What was built

The **Support@ mode** of the Unified Inbox. When Andy selects a `support@` thread, a ticket overlay layers above UI-8's single-thread detail: type chip (Haiku-classified + manually re-pickable), status pill (Open / Waiting on customer / Resolved), Close-ticket action inside the composer, and a right-side collapsible Customer Context panel. UI-10 also replaces the two UI-8 stubs — attachment upload via Graph (download-only became full drag-drop/click-upload), and calendar-invite RSVP via Graph (stub toast became real Accept / Tentative / Decline wired through `rsvpCalendarInvite`).

**Architectural pivot at G1 locked silently** (`feedback_technical_decisions_claude_calls`): the brief specced a brand-new `inbox_support_tickets` table to avoid the B1 `support_tickets` naming collision. Actual `threads` schema already carried `ticket_type`, `ticket_status`, and `ticket_resolved_at` columns from a prior spec alignment, so UI-10 **extended `threads` in place** (added two new columns: `ticket_type_assigned_by`, `ticket_resolved_at_ms`) rather than introducing a second tickets table. Cleaner, removes the 1:1 FK and the enum-column duplication, keeps ticket state on the same row as thread state — aligns with spec §16 #56 (thread-level aggregate).

## Key decisions locked (all silent per `feedback_technical_decisions_claude_calls`)

1. **Extended `threads` in-place, not new `inbox_support_tickets` table.** Brief called for a separate table to dodge the B1 `support_tickets` naming collision. On G1 inspection, `threads` already carried `ticket_type` + `ticket_status` + `ticket_resolved_at` from a prior Phase 3.5 alignment, so the second table would have been a duplicate 1:1 wrapper. Added the two missing columns (`ticket_type_assigned_by` + `ticket_resolved_at_ms`) via migration 0037 and let ticket state live on the thread row. Matches §16 #56 (thread-level aggregate) cleanly — no FK dance, no enum-column duplication. B1's `support_tickets` (reportIssue / bug report surface) stays untouched as a fully separate concept.
2. **`writeTicketStatus` is a single shared helper.** `setTicketStatusAction` + `closeTicketAction` + the worker's auto-resolve handler + the sync.ts re-open-on-inbound-reply path all route through one function in `ticket/actions.ts`. Gates on the transition (no-op if unchanged), stamps `ticket_resolved_at_ms`, logs the right activity kind, re-enqueues auto-resolve when re-opening. Avoids drift between four call sites writing the same three columns differently.
3. **Inline `sendMail` attachments, not upload-session.** Graph's `POST /me/sendMail` + `{ attachments: [{ "@odata.type": "#microsoft.graph.fileAttachment", contentBytes: base64 }] }` works end-to-end for files ≤3MB. Larger files need Graph's `createUploadSession` flow (chunked PUTs). v1.0 accepts the 3MB cap; the upload-session path is a clean additive patch for a future attachments-polish session. Client-side + server-side both validate so the 3MB edge is a friendly error, not a cryptic 413.
4. **Attachments held in client state, flushed through sendCompose.** No separate `uploadAttachmentToDraft` server action (despite brief §4 listing one). The ReplyComposer's local state holds the `File[]` until Send / Close-ticket; `sendCompose` / `closeTicket` base64-encode on submit, pass to Graph, log the `inbox_attachment_uploaded` activity. Simpler than a per-draft upload/revoke dance against Graph's draft-attachment endpoints, and drafts are thin enough that client-state attachment persistence is fine across navigations (re-uploaded by Andy if they navigated away mid-compose). If future drafts-autosave lands (`ui_6_autosave_for_compose_drafts`), attachment persistence to `compose_drafts.attachments_json` becomes a natural extension.
5. **New invite *creation* in compose modal deferred to PATCHES_OWED.** The compose modal's Calendar Invite button (UI-8 left disabled with "UI-10" tooltip) is wired for RSVP but not for creating new invites from scratch. Creating a new invite needs a date/time picker + attendee list + room/location input + Graph's `/me/events` endpoint — that's a full sub-feature that doesn't belong in the ticket-overlay scope. RSVP to existing invites is the 80% use case Andy's actually hit in production Graph sync; creating invites from Lite (vs. Outlook / Calendar) is a later compose-polish nice-to-have.
6. **Re-open on inbound reply to resolved ticket.** Spec §4.3 says "on inbound reply from customer → 'Open'" but doesn't explicitly cover the resolved-then-replied case. Interpretation: if a customer emails back after we resolved their ticket, the ticket re-opens (not a separate new ticket). Implemented in `sync.ts` — on new inbound on a support@ thread with `ticket_status === 'resolved'`, flips to `'open'` and re-enqueues auto-resolve. Same thread row; activity log records the re-open. Matches "it's not done until they agree it's done" product feel.
7. **Classifier runs opportunistically on sync, not in a dedicated cron.** The Haiku classifier fires from `sync.ts` on any support@ thread with `ticket_type === null` when a new inbound arrives. No separate backfill cron — existing untyped threads get classified on their next inbound. Trade-off: a support@ thread that stopped receiving inbounds before UI-10 shipped stays untyped until Andy opens it and manually picks a type. Acceptable because (a) untyped threads show "Classifying…" placeholder which is non-blocking UX, (b) the classifier cost is bounded by natural inbound rate.
8. **Customer Context panel reuses patterns, no new data model.** Reads straight from `contacts` + `companies` + `deals` + `saas_subscriptions` + `activity_log`. No new tables, no CCE call (CCE-1 hasn't shipped; when it does, the panel upgrades to read from CCE's richer snapshot per `project_two_perpetual_contexts`). For today: plain SQL, plain prose, falls back gracefully when fields are missing.
9. **Close-ticket button lives in the composer, not the overlay.** Spec §4.3 says Close-ticket is "alongside Send in the composer" — so it's a composer-level button gated by `closeTicketEnabled={thread.sending_address === 'support@' && thread.ticket_status !== 'resolved'}`. Overlay chrome stays read/toggle only (type chip + status pill). Keeps the hero action (Send / Close) in one place, which matches `feedback_primary_action_focus` — no split primary actions between overlay and composer.
10. **Optimistic state with rollback on server error.** Type chip + status pill + RSVP buttons all apply the selection locally first, call the server action, and roll back the local state if the action throws. Matches UI-8's conventions (composer Save/Discard are similarly optimistic). Feels instant even on a cold Graph round-trip. `console.error` logs go to the operator console; silent rollback avoids a noisy toast for transient flaky Graph calls.
11. **Sparkles cue, not a label.** The brief's "Claude-assigned" vs "Andy-assigned" signal is carried by a `lucide-sparkles` icon next to the type label when `assignedBy === 'claude'`, suppressed when `assignedBy === 'andy'`. No "AI-picked" text label — dry, observational, consistent with `feedback_no_lite_on_client_facing` and the brand's anti-"synergy" tone. Hover title gives the full sentence.
12. **Auto-resolve threshold in `settings`, not a code constant.** `inbox.ticket_auto_resolve_idle_days = 7` lives as a seeded settings row (migration 0037). The handler reads it on every fire. Matches `project_settings_table_v1_architecture` (v1.0 data, v1.1 UI). If Andy wants to tune the idle window post-launch, it's a one-row UPDATE, not a code release.

Mobile has to compress: the overlay bar can fit at the top of the thread detail view, but the Customer Context panel becomes a bottom sheet or expandable drawer triggered from a button in the overlay. UI-11 must decide whether the panel is reachable at all on mobile (probably: yes, via a sheet) or collapses to a summary chip with tap-to-expand.
- **Composer attachment + RSVP buttons.** Attachment drag-drop doesn't exist on mobile; a file-picker trigger is the whole UX. RSVP buttons stay identical. Both components already handle touch (no mouseenter-only affordances).
- **Optimistic state + rollback pattern.** Type chip + status pill + RSVP buttons all implement optimistic local state with rollback on error. UI-11 mobile interactions inherit this directly — the same components render.
- **Sibling-render vs overlay positioning.** UI-9 locked sibling-render for composer preservation on desktop. UI-11 needs to decide: mobile thread detail is full-screen, so the ticket overlay layers above within the single full-screen view. Composer preservation still matters — re-use the sibling-render pattern within the mobile thread-detail surface.
- **Voice surfaces intact.** All UI-10 voice copy ("Classifying…", "Who they are, where they are.", RSVP confirmations) works unchanged on mobile — shortened responsive variants only if space-constrained, per `feedback_no_content_authoring`.
- **Offline caching scope.** UI-11 needs to decide whether ticket status changes (status pill) are allowed offline with sync-on-reconnect. Risk: stale status writes could clobber server state. Safest v1.0: disable overlay interactions when offline, show banner "Offline — ticket actions pause."
